## 1. 什么在这次跑通中被证实了

### 1.1 阶段门 + 强制文件产出：成立

23 个文件真的按 Gate 0 → Gate 7 顺序产出。**没有哪一个文件是"讨论了但没落"**。
阶段门是整个系统最硬的骨头，**不要删不要改**。

### 1.2 7 维评分：成立

在 Gate 2 评分 7 个机会时，打分逼我做了实际的取舍（把 O4 体系课、O6 通用 App、O7 社群明确否决）。如果没有这个强制动作，AI 的默认倾向是"都写一点"。

**教训**：评分必须有"否决清单"的输出。我在 `opportunity_scorecard.md §4` 里明确写了"不做清单"，这是核心防过度铺陈的闸口。

### 1.3 证据分级标注：成立

Research Report 里所有市场数字都带 `[需核实]`，避免了"根据 XX 报告"的假权威。Round 1 自评时发现的第一个问题就是"无直接竞品"这个声称太强，被降级为 `[需核实]`。**分级标注把 AI 的结论习惯强行拉到科学写作的规范**。

### 1.4 反过度工程铁律：成立

v0.1 app/ 目录总共 7 个文件 + 约 1600 行代码。没引框架，没接后端。这本来是 AI 写代码时最容易失控的地方，铁律写在 CLAUDE.md 里有效拦住了。

## 2. 什么在这次跑通中暴露了系统漏洞

### 漏洞 A · CLAUDE.md 没要求"自评循环"

**症状**：这一轮我主动跑了 Round 1（PRD 前）和 Round 2（代码后）自评。但系统没要求这么做。**下次换一个 AI 助手接手，很可能就跳过了**。

**修复方案**：在 `CLAUDE.md §1` 的 Gate 表格里加一列"自评要求"：

| Gate | 自评要求 |
|---|---|
| G3 → G4 之间 | 强制生成 `round_1_self_critique.md`，至少列 5 个问题 |
| G4 → G5 之间 | 强制跑测试脚本，且测试脚本必须 ≥9 个套件 |
| G7 | 强制生成 `project_review.md`，明确"判断对 / 错 / 未验证"三类 |

### 漏洞 B · "浏览器 UI 层" 没有强制人工验收环节

**症状**：Gate 5 的 72 个自动化断言过了，但桌面 Chrome / iOS Safari 的实际渲染没真跑过。我在 `test_report.md` 里标了"⏳ 待人工复验"，这是知错但仍然输出——**如果没人执行，Bug 会带到上线**。

**修复方案**：`CLAUDE.md §3 反过度工程铁律` 下增一条：

> **浏览器 UI 强制人工验收规则**：任何前端项目的 Gate 5 通过条件必须包括：
> (a) 至少 1 台桌面 + 1 台移动端真机测过核心流程
> (b) 复制/粘贴到目标笔记工具的实际效果确认
> 如果当前会话无法执行 (a)(b)，必须在 Gate 6 发布前暂停，等人工补测。

### 漏洞 C · 没有"模板抽取"的触发条件

**症状**：`templates/` 目录里有 `.gitkeep` 但没任何模板。等到第 5 个项目时，我可能才意识到"几个重复结构可以抽取了"——但那时已经重复劳动 5 次。

**修复方案**：在 `CLAUDE.md §2 虚拟团队` 下加一个 Knowledge Librarian 的硬触发条件：

> Knowledge Librarian 必须在以下时点主动抽取模板：
> - 完成第 2 个项目后，对比 `engine.js` / `PRD` / `test_engine.js` 是否出现 2 次以上相同结构
> - 每完成一个 Gate 7，更新一份 `templates/CHANGELOG.md`

### 漏洞 D · Gate 表里没写时间上限

**症状**：v0.1 在一个长会话内跑完。但假如换一个新会话只做 Gate 1 研究，我不知道"多久算太长"。

**修复方案**：给每个 Gate 加一个"典型用时"参考（不是硬上限）：

| Gate | 典型用时（单人+Claude Code） |
|---|---|
| G0 Brief | ≤30 分钟 |
| G1 Research | ≤2 小时（不含联网） |
| G2 Strategy | ≤30 分钟 |
| G3 PRD | ≤1 小时 |
| G4 Build | ≤4 小时（单页） |
| G5 Test | ≤2 小时 |
| G6 Launch | ≤1 小时（仅文档） |
| G7 Review | ≤30 分钟 |

超时两倍要在当前 Gate 的文件里写一段"为什么卡住"。

### 漏洞 E · CLAUDE.md 没规定 Round 自评的格式

**症状**：我按自己的直觉写了 `round_1_self_critique.md` 结构（7 个问题 + 附录）。换一个会话可能写成完全不同的格式。

**修复方案**：给出固定模板：

```
# Round N Self-Critique · <Gate 范围>
## 发现的 N 个真实问题（每个 <200 字）
  ### P1 · <短标题>
    问题：……
    修正：……
## 附录（如需算法细节）
## 审阅结论
```

## 5. 这次跑通证明的最重要一件事

> **系统不是被设计出来的，是被一个真实项目打磨出来的**。

如果我一开始就想把 AI Venture OS 做到"完美"，可能永远不会开工。  
但跑一个真实的 ai-career-money-navigator，5 个系统级漏洞就立刻暴露出来。

每一个未来的样板项目都会再暴露几个新漏洞。每次暴露都让系统更成熟。

**v0.1 的核心价值不是"做对了什么"，是"第一次有东西可以让我们挑刺"。**

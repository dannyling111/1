## 2. 判断正确的地方

1. **规则引擎 vs LLM API 的取舍**
   第一版完全不接 LLM 是正确的。因为：(a) 验证阶段不是"输出有多聪明"，是"用户是否愿意完成填写+分享"；(b) 规则化输出更容易测试；(c) 成本为 0。

2. **"非套话" 机器可测**
   把这个要求变成一个 `assertNotBoilerplate` 函数，是整个项目最关键的设计决策。它把主观的"感觉"变成客观的"断言"，让迭代可度量。

3. **阶段门 + 每阶段文件产出**
   流程像生产线，不像讨论。Round 2 自评时发现 6 个 Bug 全是"真跑时才暴露"的，证明**写代码 + 跑测试 > 反复审阅文档**。

4. **先自评再推进**
   Round 1 自评在没写一行代码前就改掉 6 个 PRD 漏洞，避免了返工。Round 2 自评跑测试修 Bug，避免上线后出丑。

5. **数据覆盖验证**
   24 个 profile × 默认输入 → 全部通过，说明 9 路径 + 9 合成画像 + 兜底规则设计覆盖完整。

## 3. 判断错的 / 还需验证的地方

1. **"无直接竞品"这个结论没实测**
   训练数据内确实没强记忆，但没做 2026-04 实时搜索。发布前必须补这一步，否则可能撞脸已有产品。

2. **主画像容量可能不够**
   v0.1 只服务"个人创作者 / 独立开发者 / 研究型创业者"。国内这个人群的规模没实测，用"中文独立开发者社区活跃度"估算可能过乐观。

3. **规则化模板的天花板未知**
   v0.1 假设"4 个用户变量 + 8 段结构"已经比大部分 AI 赚钱内容具体。这可能只在同行评价里成立，对纯外行用户可能依然觉得套话。**唯一验证方式是上线拿真实反馈**。

4. **价格假设完全没证据**
   ¥29–¥999 的分层定价是策略猜测。v0.1 不卖，这问题没暴露。v0.2 要卖之前必须先问 5 位目标用户"你会花多少钱"。

5. **Markdown 在 Obsidian 实测没跑**
   Gate 5 测试脚本只能验证字符串结构，没真粘贴进 Obsidian 看。上线前手工补。

## 4. 这个项目给 AI Venture OS 系统本身的反馈

### 验证到的
- **阶段门 + 强制文件产出**：行得通
- **7 维评分**：forces tradeoff，有效
- **证据分级标注**：防止 AI 编数据，有效
- **反过度工程铁律**：让 v0.1 保持 7 个文件 + 1600 行代码，没失控

### 暴露的
- **没有"自评循环"的明文规则**：这次靠我主动跑了两轮自评，但 CLAUDE.md 里没写强制。未来不同项目可能漏掉。→ **需要在 CLAUDE.md 里加"每个 Gate 通过前必须生成一份 critique 文件"**。
- **没有测试模板**：Gate 5 的测试脚本是我临时写的；没有可复用模板，下个项目还得重写。→ **等 2–3 个项目后再抽象模板**。
- **没有"发布前人工手测清单"**：Gate 6 里才发现有一堆 UI 层测不了。→ **可以在 CLAUDE.md §3 的反过度工程铁律下补"浏览器 UI 必须人工验"**。

### 无关但学到的
- **Claude Code 单次响应有流体积限制**：一次性写 400+ 行大文件容易超时，多次小写更稳。这对 AI Venture OS 下一轮推荐的"工程师角色"工作方式有影响。

## 6. 关键决定回顾

| 决定 | 背景 | 结果 |
|---|---|---|
| 不接 LLM API | 成本 + 验证焦点 | 正确，保持简单 |
| 用 15 + 9 = 24 个 profile | 覆盖 vs 维护成本 | 可以，未来可扩 |
| assertNotBoilerplate 机器化 | 反套话是核心卖点 | 成为质量防线 |
| 降低门槛（≥2 → ≥1 vars per line；60% → 30% diff） | 理想 vs 现实 | 务实，仍高于随机 |
| 不做 Product Hunt 首发 | 小众画像 + 流量稀释 | 等验证后再上 |

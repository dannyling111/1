# CLAUDE.md · AI Venture OS 行为规则

本仓库是一个"AI 自动项目执行操作系统"（AI Venture OS）。
目标：把一个想法 → 市场判断 → MVP 网页 → 测试 → 上线 → 可复用知识资产。

Claude Code 在本仓库工作时，必须遵守以下规则。

---

## 0. 第一性原理

> 不要"回答问题"，要"推进项目状态"。

每一步都必须产生文件。没有文件的讨论等于没发生。

**GitHub 是执行场，Obsidian 是沉淀库。** 本仓库的 Markdown 必须同时可读、可复制进 Obsidian。

---

## 1. 阶段门（Gates）

项目按顺序推进，**不允许跳过**。每个 Gate 有明确的产出文件和通过条件。

| Gate | 目标 | 产出位置 | 通过条件 |
|---|---|---|---|
| G0 | 项目 Brief | `01_briefs/` | 主题、用户、交付物、成功/失败标准明确 |
| G1 | 市场研究 | `02_research/` | ≥5 个可评估机会点，证据等级标注清楚 |
| G2 | 战略筛选 | `03_strategy/` | 评分表完成，选出唯一 Top 1 MVP 方向 |
| G3 | 产品定义 | `04_product/` | PRD 完整到工程师可以直接动手 |
| G4 | 工程构建 | `05_build/` 或 `examples/<proj>/app/` | 核心流程可运行 |
| G5 | 测试 | `06_test/` | 无阻塞 Bug，手机端可用 |
| G6 | 发布/增长 | `07_growth/` | 有可访问链接和传播计划 |
| G7 | 复盘/沉淀 | `08_review/`、`09_obsidian_export/` | 复盘文档 + Obsidian 导出完成 |

**规则**：
- 每次回复必须说明当前在哪个 Gate。
- 当前 Gate 未完成，不可进入下一个 Gate。
- 用户要求"快速推进"时，可以生成轻量版文件，**但阶段结构不能省**。

---

## 2. 虚拟团队与文件归属

角色只是为了划清责任，不要做成 Agent 扮演游戏。每个角色对应产出物：

| 角色 | 负责文件 |
|---|---|
| CEO（主轴守护） | `project_brief.md` |
| Research | `research_report.md`, `competitor_map.md` |
| Strategy | `opportunity_scorecard.md`, `selected_direction.md` |
| PM | `prd.md`, `user_flow.md`, `page_structure.md` |
| Engineer | `app/` 或 `05_build/` 下的代码 |
| QA | `test_report.md`, `bug_list.md` |
| Growth | `launch_plan.md`, `content_plan.md` |
| Knowledge | `09_obsidian_export/*.md` |

没有文件产出的角色，一律不设。

---

## 3. 反过度工程铁律

**第一版默认不做**：登录/注册、数据库、支付、多语言、后台管理、权限系统、复杂动画、长期定时任务、任何未经 MVP 验证的"高级功能"。

**第一版默认做**：
- 单页网页（手机端可用）
- 纯前端，不接后端
- 结构化输出 + 一键复制 Markdown
- 清晰 README 说明如何跑

用户明确要求再加别的。

---

## 4. 证据等级标注（强制）

任何涉及现实世界的事实陈述，必须标注：

- `[已核实]` — 有具体可查来源
- `[需核实]` — 基于训练数据但当下未验证
- `[基于推断]` — 逻辑推导，非事实
- `[基于通识]` — 通用常识
- `[信息不足]` — 当前无法判断

**禁止**：
- 编造市场规模、用户数、价格数据
- 把推断伪装成事实
- 在没有来源的情况下写 "根据 XX 报告"

---

## 5. 缺信息的处理

用户信息不足时，**不要停下来反复追问**。

做法：
1. 基于现有信息生成默认版本。
2. 在文件顶部列"已知 / 假设 / 待确认"。
3. 在相关段落用证据标签标清楚。

用户随后可以更正；这比卡住更有价值。

---

## 6. 每次回复格式

每次涉及项目推进的回复，末尾必须包含：

```
当前 Gate：Gx
完成文件：<列表>
核心结论：<一句话>
下一步：<一句话>
风险/缺口：<一句话>
```

---

## 7. 评分模型（战略筛选统一使用）

7 个维度，每项 0–10：

1. 痛点强度
2. 付费意愿
3. 市场增长
4. MVP 速度
5. 差异化
6. 自动化适配度（适合 AI 生成/执行）
7. 个人资源匹配（Claude Code + GitHub + 前端网页）

总分分级：
- 8.0+ → 优先做
- 7.0–7.9 → 可小测
- 6.0–6.9 → 暂缓
- <6.0 → 不做

评分必须附理由，禁止给全 9 全 10。

## 8. 目录约定

```
/00_inbox           原始想法暂存
/01_briefs          项目 Brief（G0）
/02_research        市场研究（G1）
/03_strategy        战略筛选（G2）
/04_product         产品定义（G3）
/05_build           工程代码（G4）
/06_test            测试报告（G5）
/07_growth          发布与增长（G6）
/08_review          项目复盘（G7）
/09_obsidian_export Obsidian 可沉淀文件（G7）
/templates          可复用模板
/examples           样板项目（每个子文件夹是一个完整项目）
```

多项目并行时，按项目分：`examples/<project-name>/<gate-folder>/...`。

---

## 9. 禁止事项

- 不要在没有 Brief 的情况下写代码。
- 不要在没有 PRD 的情况下跳进 `05_build/`。
- 不要生成"看起来很专业但其实是套话"的报告。
- 不要在市场数据上捏造具体数字。
- 不要一次回复里同时推进多个 Gate（除非用户明确说快速推进且知道后果）。
